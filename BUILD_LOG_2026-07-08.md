**Key Components** (in `experiments/test_adversarial_signed_envelope.py`, 36 tests across 7 classes):

1. **TestT1PayloadTampering (8 tests)** — payload-level attacks:
   - `test_zero_byte_replacement` — replace one char in payload, expect PAYLOAD_MISMATCH
   - `test_full_byte_flip` — XOR every byte of payload, expect PAYLOAD_MISMATCH
   - `test_missing_required_key` — drop a required key, expect PAYLOAD_MISMATCH
   - `test_missing_list_item` — drop a list item, expect PAYLOAD_MISMATCH
   - `test_type_confusion_string_to_int` — change a string field to int, expect PAYLOAD_MISMATCH
   - `test_type_confusion_int_to_string` — change an int field to string, expect PAYLOAD_MISMATCH
   - `test_deep_mutation` — mutate a nested field, expect PAYLOAD_MISMATCH
   - `test_empty_payload_attack` — sign then submit empty payload, expect PAYLOAD_MISMATCH

2. **TestT2EnvelopeTampering (5 tests)** — envelope-level attacks:
   - `test_signature_byte_flip` — XOR first byte of signature (urlsafely), expect INVALID_SIGNATURE
   - `test_swap_signature_from_other_envelope` — replace signature with a valid signature from a different payload, expect INVALID_SIGNATURE
   - `test_key_id_swap` — replace key_id with a different registered key, expect INVALID_SIGNATURE
   - `test_algorithm_swap` — change HMAC-SHA256 → ed25519 declared, expect INVALID_SIGNATURE
   - `test_timestamp_skew` — mutate `issued_at` to a different value, expect INVALID_SIGNATURE (the signed_bytes include `issued_at`)

3. **TestT3AlgorithmSubstitution (3 tests)** — algorithm confusion:
   - `test_hmac_to_ed25519_declared_swap` — claim envelope is ed25519 but signed with hmac, expect INVALID_SIGNATURE
   - `test_ed25519_to_hmac_declared_swap` — claim envelope is hmac-sha256 but signed with ed25519, expect INVALID_SIGNATURE
   - `test_malformed_algorithm_string` — set `algorithm = "not-a-real-algo"`, expect UNKNOWN_ALGORITHM

4. **TestT4KeyCompromise (3 tests)** — key compromise / rotation:
   - `test_key_removed_from_registry_after_signing` — sign, then `unregister`, expect UNKNOWN_KEY on verify
   - `test_key_added_after_signing_re_verifies` — sign with one key, then register a different one, original envelope still verifies (deterministic key-id binding)
   - `test_key_reregistered_with_different_material` — re-register a key with different bytes, expect INVALID_SIGNATURE for the old envelope (key-id binding to material is enforced)

5. **TestT5ReplayAndFreshness (5 tests)** — replay + freshness:
   - `test_not_before_in_future` — envelope with `not_before=now+1000`, expect NOT_YET_VALID
   - `test_expires_at_in_past` — envelope with `expires_at=now-1000`, expect EXPIRED
   - `test_time_tolerance_window` — verify exactly at `expires_at`, expect EXPIRED (boundary is exclusive on the right)
   - `test_no_not_before_always_valid` — envelope without `not_before`, never NOT_YET_VALID
   - `test_no_expires_at_never_expires` — envelope without `expires_at`, never EXPIRED

6. **TestT6ShapeAttacks (5 tests)** — envelope-shape confusion:
   - `test_cosign_without_secondary_signature` — claim COSIGN but no `cosign_signature`, expect MALFORMED_ENVELOPE
   - `test_cosign_without_secondary_key_id` — claim COSIGN but no `_secondary_key_id` in metadata, expect MALFORMED_ENVELOPE
   - `test_detached_with_embedded_payload` — claim DETACHED but `payload` is non-None, expect PAYLOAD_MISMATCH (recomputed digest on payload)
   - `test_envelope_with_none_payload` — claim ENVELOPE but `payload=None`, expect MALFORMED_ENVELOPE
   - `test_cosign_with_hmac_secondary` — primary HMAC, secondary Ed25519, both signatures valid → VALID; flip one byte in either → INVALID_SIGNATURE

7. **TestNonAttacks (7 tests)** — sanity / non-attack regressions:
   - `test_legitimate_key_rotation` — sign with key1, register key2, both verify; `unregister` key1, old envelope → UNKNOWN_KEY
   - `test_identical_payload_identical_digest` — same payload → same `payload_digest` (deterministic content addressing)
   - `test_fresh_resign_after_tamper` — if attacker tampers, re-sign with same key, new signature is different (each signing is fresh)
   - `test_deterministic_signing_bytes` — same payload + same key + same timestamp → same `signed_bytes`
   - `test_schema_version_preserved` — `schema_version` survives the round-trip
   - `test_large_payload_10kb` — sign + verify a 10KB payload, expect VALID
   - `test_nested_components` — payload with nested `components[].verb` list, expect VALID

**Threat model coverage**: 36 tests cover 7 distinct threat-model classes. The substrate rejects every tampered envelope with a *named* `VerificationStatus` (INVALID_SIGNATURE / PAYLOAD_MISMATCH / UNKNOWN_ALGORITHM / UNKNOWN_KEY / MALFORMED_ENVELOPE / EXPIRED / NOT_YET_VALID). No silent acceptance. No false negatives.

**Adversarial findings** (the substrate is correct):
- **T1 (payload tampering)**: 8/8 rejected. The `payload_digest` is the canonical hash; any byte change produces a different digest, the signature doesn't match.
- **T2 (envelope tampering)**: 5/5 rejected. The `signed_bytes` include `key_id`, `algorithm`, `issued_at`, `not_before`, `expires_at`, `shape`, `payload_digest`, `payload` (if present), `metadata` (if present). Mutating any of these changes the signed bytes; the signature doesn't match.
- **T3 (algorithm confusion)**: 3/3 rejected. The verifier dispatches on the declared `algorithm`; using a different key type with the declared algorithm produces an invalid signature.
- **T4 (key compromise)**: 3/3 as expected. The substrate's `KeyRegistry` is a thin dict; `unregister` removes the key, the verifier returns UNKNOWN_KEY.
- **T5 (replay)**: 5/5 as expected. The envelope carries `not_before` and `expires_at`; the verifier enforces both.
- **T6 (shape confusion)**: 5/5 rejected. The verifier inspects the shape field and enforces its requirements.
- **Non-attacks**: 7/7 as expected. The legitimate flows (key rotation, fresh re-sign, deterministic, large payloads) all work.

**Two real bugs surfaced and fixed** (in the test, not the substrate):
1. **base64 alphabet mismatch**: the test used `__import__("base64").b64decode` (standard alphabet) on a `urlsafe_b64encode`-generated signature. Fixed to `urlsafe_b64decode`. The substrate was correct; the test was wrong. This is a *test* bug, not a *substrate* bug — but it's worth noting because the substrate's docstring should call out the urlsafe alphabet more prominently.
2. **`KeyRegistry.revoke` doesn't exist**: the test assumed a `revoke` method; the substrate exposes `register` / `unregister` / `resolve`. Replaced `revoke` with `unregister`. The substrate's API surface is *deliberately minimal*: a registry is a dict, nothing more. Production deployments should use a vault-backed resolver with their own revocation policy.

**Research synthesis**:
- **The agent-gateway era is here**. The week's research converges on a single architectural shape: a control plane sits between agent and tool, every tool call passes through the plane, the plane emits an *advisory* that downstream consumers (gateways, CI gates, OPA evaluators, human reviewers, AIOps platforms) act on. Forbes (Jul 5), Nutanix, Arcade on Azure/AWS, HashiCorp Terraform MCP, InnoviumAI, HarnessBridge, ToolBench-X, PagerDuty, the AIRA Humanity & AGI Summit at Stanford (Jul 12 2026) — all describe the same shape.
- **The signed envelope is the advisory wire format**. Three layered claims:
  1. *Content integrity*: signature is over the canonical hash; any byte change invalidates.
  2. *Authenticity*: signature is over the `key_id`-bound signing bytes; provenance is the key.
  3. *Freshness*: `not_before` / `expires_at` give the consumer a refusal criterion beyond signature validity.
- **The 36 adversarial tests demonstrate all three claims**. 7 threat-model classes, every tampered envelope rejected with a named `VerificationStatus`. No silent acceptance. No false negatives.
- **The `KeyRegistry` is deliberately minimal**. Register, unregister, resolve. No `revoke`, no TTL, no audit log of lookups. The substrate's API surface is a *contract* with the operator: "if you want a richer policy, bring your own resolver". Vault-backed resolvers, HSM-based signing, FIPS 140-2 modes — all are operator concerns, not substrate concerns.
- **The substrate cannot prevent malicious signing by a compromised key**. This is by design: the substrate trusts the operator to manage key material correctly. The adversarial test for T4 (key compromise) is about *post-signing* rotation, not *pre-signing* compromise. The substrate's design says so explicitly.
- **Connection to research signals**:
  - *ToolBench-X* hazard types (5) → envelope's status codes (5+). The substrate already names what ToolBench-X is trying to benchmark.
  - *Multi-step RL collapse* → envelope's content addressing survives the model swap. Audit trails outlive the generator.
  - *RedAct watermarks* → envelope's `metadata` field is the natural carrier. A signed watermark + a signed envelope = double-attested provenance.
  - *HarnessBridge action projection* → the projection can refuse to emit an unsigned envelope. Signed-by-default is a substrate property.
  - *VLM Self-Audit* → same pattern: actor emits signed metadata, auditor verifies. The envelope is the substrate, the audit is downstream.
  - *PagerDuty AIOps* → "agent and model drift show up differently than a conventional software crash". The envelope is the *signal* the AIOps would alert on. An unsigned or unverified advisory IS the drift signal.
