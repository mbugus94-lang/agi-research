## Build Task: Memory Refiner (MemRefine-inspired compression/refinement layer)

### Motivation

`TieredMemorySystem` (L0/L1/L2) handles the storage and consolidation side of memory, but its eviction policy is purely importance-decay. MemRefine (arXiv:2606.13177v1) makes a sharper claim: similarity should only *propose* candidate pairs; the actual KEEP / MERGE / COMPRESS / EVICT / PROMOTE decision should be made by a judge that reasons over factual content. Today's build implements that split for our agent memory.

### Key Components

1. **`RefinementAction`** — KEEP / EVICT / PROMOTE / COMPRESS / MERGE
2. **`RefinementCandidatePair`** — similarity-proposed pair, contains both contents + metadata; judge never sees similarity as a verdict signal
3. **`RefinementDecision`** — judge's verdict with action, target_ids, reason, judge name, confidence, optional compressed_content, decided_at timestamp
4. **`RefinementJudge` (Protocol)** + **`FactDensityJudge`** (default deterministic heuristic) + **`LLMJudgeStub`** (records calls for testability; replaceable with real LLM)
5. **`RefinementReport`** — initial_size, final_size, iterations, evicted, compressed, merged_pairs, promoted, kept, decisions[]
6. **`MemoryRefinerConfig`** — similarity_threshold, max_iterations, max_pair_scan, audit_path, promote_tier, protect_above_importance
7. **`MemoryRefiner`** — orchestrator with: `propose_pairs()`, `judge_pair()`, `refine(budget)`, `compress_one(entry_id)`, `evict_one(entry_id)`, `merge(a_id, b_id)`, `summary()`
8. **`_compress_facts` / `_merge_facts` / `_similarity` / `_fact_density` / `_novelty_score` / `_char_jaccard` / `_cosine_similarity`** — pure helper functions, all deterministic, no LLM calls
9. **`create_memory_refiner(memory, judge, audit_path)`** — 1-line install

### Conservative Posture (the invariants)

- **`compress_one` always returns a COMPRESS decision** (the name is a promise; the operator chose compression, not eviction)
- **`evict_one` refuses to evict above `protect_above_importance`** (default 0.85; the operator's high-importance floor is hard)
- **The judge never auto-promotes an entry**; promotion is a deliberate decision separate from compression/eviction
- **The judge never silently merges two high-importance entries** (both ≥0.7 ⇒ KEEP; merge would risk losing signal)
- **Merge uses importance-delta ≥ 0.3** as a one-high-one-low signal; below that, KEEP both
- **Pair proposals are bounded by `max_pair_scan`** (default 200) to keep O(n²) cost predictable
- **The audit trail is append-only** — `MemoryRefiner.audit` is a list, never truncated by the refiner itself
- **`audit_path`** (optional) persists every decision to JSONL — the disk is the system of record
- **The refiner never mutates the underlying `TieredMemorySystem` outside its own public API** (it uses `_remove_entry` / `_replace_entry_content` wrappers, not direct dict surgery)
- **Budget-pressure escalation**: when `current_size > budget`, the FactDensityJudge escalates medium-fact-density / low-importance entries to EVICT (otherwise the loop would converge on compress-only and never reduce size)
- **Convergence check**: the loop terminates when no size-changing action (EVICT or MERGE) is taken in an iteration (compression alone doesn't reduce size and shouldn't trigger another iteration)
- **The judge is LLM-agnostic** — `LLMJudgeStub.always_action` is a test affordance, not a runtime escape hatch

### Research Synthesis

- **MemRefine's central insight is the proposal/verdict split**. Similarity is noisy; factual content is the signal. Our `RefinementCandidatePair` carries the similarity for *proposal*; the judge never sees it as a verdict signal
- **The budget-pressure escalation is the novel engineering decision**. Pure MemRefine would iterate compress-only forever in tight budgets; we add an over-budget branch in `FactDensityJudge.judge_singleton` that escalates low-importance entries to EVICT. This is the same posture as `ToolPrivilegeGovernor`'s blocklist floor: the operator's budget is a hard constraint, not a soft suggestion
- **`compress_one` always-COMPRESSES is a contract**. The operator chose compression; we don't second-guess. Mirrors the `compress_one` API in MemRefine (the paper's algorithm forces a compression verdict when called)
- **`protect_above_importance` mirrors `ToolPrivilegeGovernor.protect_above` and `SafetyCircuitBreaker`'s importance-aware gating**. The "high-value entries are never silently lost" invariant is now a repo-wide pattern
- **The audit trail is the verifiable substrate**. Combined with `ProofCarryingActionBridge.IEEC` (Jun 13), the agent's memory refinements and its actions are both on disk. Dapr 1.18's "Verifiable Execution" pattern, in-process
- **The LLMJudgeStub is the testability hook**. Recording `calls` lets tests assert what *would* be sent to a real LLM. In production the stub is replaced with a real LLM call; the contract is identical
- **The 138 tests are intentionally small and self-contained**. Each test verifies a single invariant; the full suite runs in <1s. Same posture as `EvoMaster`'s "~100 LOC to deploy" discipline
- **The module is ~1100 lines** (memory_refiner.py). Small enough to read in one sitting, large enough to be non-trivial. Matches the repo's "small, focused, working" discipline
- **The Curation-Bench finding (agents tune existing policy variants rather than explore new families) is a calibration signal for our own self-improvement**. When the agent refines its memory, it should not silently bias toward "merge similar" — the judge must be the source of variety. The `FactDensityJudge`'s five distinct outcomes are the variety
