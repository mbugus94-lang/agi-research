**Build Task**: Per-verb CEF guard policy — a `VerbPolicyBundle` that maps `(verb_name, verb_version) -> VerbCEFGuardConfig` with deterministic fallback, hash-chained audit log, and a CLI review mode. Built on top of the 2026-07-02 Typed-Verb CEF Guard.

**Motivation (two converging signals)**:

1. **2026-07-02 carryover**: The 2026-07-02 build landed `core/typed_verb_cef_guard.py` (GuardedVerbRuntime). The next-priority was explicit: "Per-verb guard policy (`VerbPolicyBundle` maps `(verb_name, verb_version) → VerbCEFGuardConfig`)". Today's build closes that carryover — a `pay` verb that fabricates an external obstacle is treated more strictly than a `read` verb that does the same.
2. **arXiv:2606.19390 (AIBOM/CSAF-VEX, "Execution-bound advisory automation for agentic AI")** — "enforced execution policies per component, runtime telemetry, deterministic replay, signed advisories". The `VerbPolicyBundle` is the typed-verb substrate's instance of the AIBOM pattern: each verb (component) has its own enforced policy, and the bundle records the policy provenance that a replay layer reads.

**Key Components**:

1. **`PolicyResolutionSource` (str-enum)** — `EXACT / WILDCARD / DEFAULT / NO_BUNDLE / GUARD_UNAVAILABLE`. The source the resolved config came from. Operator can query the audit log by source.
2. **`PolicyAuditEventKind` (str-enum)** — `REGISTER / RESOLVE / FALLBACK / MISS / UNREGISTER`. Mirrors the AIBOM telemetry taxonomy.
3. **`VerbPolicyEntry`** — per-`(verb_name, verb_version)` mapping: `config: VerbCEFGuardConfig + source + rationale + policy_digest` (sha256 of the canonicalized config). `looser_than_default` property reports whether the entry is more permissive than the bundle's `default_config` (operator-side warning surface).
4. **`VerbPolicyBundle`** — operator-controlled registry. Resolution order: (1) exact match, (2) wildcard `*` entry, (3) bundle's `default_config`, (4) `None` (caller fallback). Hash-chained audit log per bundle.
5. **`VerbPolicyAuditLog`** — append-only, sha256 hash-chained, replay-able. Each entry: `sequence + kind + payload + previous_hash + entry_hash`. `verify_chain()` re-walks and returns `(ok, first_bad_sequence)`. JSONL mirror to disk for offline replay.
6. **`BundledVerbRuntime`** — wraps a `VerbRuntime` + a `VerbCEFGuard` resolved per call from the bundle. Each `invoke`:
   - Resolves the policy for the call
   - Records a `RESOLVE` / `FALLBACK` / `MISS` audit event
   - Runs the inner verb
   - Inspects the output with the per-call guard
   - Returns a `BundledVerbStepResult(result=..., inspection=..., resolution=..., guard_action=...)`
7. **`BundledVerbStepResult`** — exposes `result`, `inspection`, `resolution`, `guard_action`, `success`, `halted`, `to_dict()`.
8. **`cli/policy_review.py`** — read a JSONL audit log (or load a bundle via `--module func`), summarize by source/kind, show last N events. The mirror of `cli/governor_circuit.py` and `cli/calibrate.py`.
9. **`create_bundled_verb_runtime(runtime, bundle, detector, halt_on_halt)`** — one-line install.

**Conservative posture**:

**Research synthesis**:
- **AIBOM/CSAF-VEX (arXiv:2606.19390)** is the strongest published pattern for the "per-verb policy" concept. The bundle's hash-chained audit log is the runtime telemetry; the `policy_digest` is the AIBOM component-attestation hash; the JSONL mirror is the advisory-automation substrate.
- The **policy provenance is the bridge to deterministic replay** — knowing which config was applied to a given call is the precondition for replaying that call and reproducing its behavior. The audit log makes that bridge observable.
- The **wildcard policy (`verb_version="*"`)** is a force multiplier for operators. A single wildcard entry can cover an entire verb family ("all versions of `pay` get the strict policy"), but the per-version pin (registration against a specific version) prevents silent policy drift across version bumps.
- **Why a per-verb `looser_than_default` flag**: A bundle that silently downgrades from a strict default to a lax per-verb override is a footgun. Surfacing it on the entry makes the operator's choice auditable. (A future CLI: `python -m cli.policy_review --warn-loose` to surface all such entries.)
- **Why hash-chain the audit log**: Tamper detection is a hard requirement for any policy-enforcement system. A `verify_chain()` method that re-walks the chain and returns `(False, first_bad_sequence)` is the substrate for "we can prove the audit log was not modified after the fact".
