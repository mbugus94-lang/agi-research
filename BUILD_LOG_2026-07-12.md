Today's research moves the substrate from "Phase 2 only" (runtime taint tracking) to "Phase 1 + Phase 2" (static MRS composition + runtime taint tracking) — closing a top-priority next-priority carryover from 2026-07-11.

- **arXiv:2607.03423 — DSCC: Dynamic Security Control Compositor (recap)**: the paper's *Phase 1* is the **MRS composition** that produces a Most Restrictive Set policy at session checkout. The substrate's `CompositionalPolicyGate` was previously only implementing the *runtime taint tracking* (Phase 2). Today's run adds the Phase 1 clearance check: every verb in a chain must have a per-tool classification >= the chain's `C_eff` (high-water mark), or the chain is rejected before any tool runs. The cluster structure is the *partition* of the policy registry by classification-level groups — the substrate's `clusters` field is a `(verb_name -> cluster_name)` map.
- **arXiv:2607.03423 §2.2 Step 3 (clearance check)**: in *clearance* mode (the paper's default), the per-tool classification must be `>= C_eff`. In *taint* mode, the check is skipped but C_eff is still recorded. The substrate now implements both modes via `CompositionMode.CLEARANCE` and `CompositionMode.TAINT`.
- **arXiv:2607.03423 §2.3 (monotonicity invariant)**: extending a chain can only tighten the result. The substrate's `chain_effective_classification` is `max(initial, max(tool.classification for tool in chain))` — monotonic by construction.
- **arXiv:2607.03423 §4.3 (results)**: 79.2% of policy *pairs* and 95.5% of *triples* are blocked under clearance mode. The substrate's `test_two_verb_mixed_classification_blocks` and `test_three_verb_mixed_classification_blocks` validate the cluster-mismatch path; the empirical block rate is *a property of the policy registry* (which verbs you register + their classifications), not a property of the gate.
- **arXiv:2607.03423 §2.2 (mode parameterization)**: the `mode` parameter (CLEARANCE / TAINT) is set at gate-construction time. The substrate's `composition_mode=` constructor parameter matches the paper's `mode ∈ {Clearance, Taint}`.

The core insight: **DSCC's Phase 1 is a *static* check (can the chain even start?), Phase 2 is a *runtime* check (did the actual data breach the policy during execution?)**. The substrate now has both phases. The `CompositionMode` enum is the operator's switch.

## Build Task: Add DSCC clearance mode to `CompositionalPolicyGate`

**Motivation**: The 2026-07-11 next-priority item was: "DSCC clearance-mode wiring (arXiv:2607.03423) — partition the policy registry into classification-level clusters; pre-compute the MRS (Most Restrictive Set) for each cluster; require the agent to declare the cluster at chain proposal time. arXiv:2607.03423 reports 79.2% / 95.5% block rate." ### What's new in the gate

#### 1. `ClassificationLevel` enum
- `PUBLIC < INTERNAL < CONFIDENTIAL < RESTRICTED` — the four-level sensitivity scale from the DSCC paper.
- `classification_rank(c)` and `max_classification(a, b)` helpers.
- `ClassificationLevel.parse(value)` accepts strings, ints, or existing `ClassificationLevel` instances.

#### 2. `CompositionMode` enum
- `CLEARANCE` (default) — per-tool classification must be >= C_eff. Stricter; matches the paper's 79.2%/95.5% block rate.
- `TAINT` — clearance check is skipped; classification is sticky on the data. The substrate's existing taint tracking handles the runtime phase.

#### 3. `DenyReason.CLEARANCE_LEVEL_INSUFFICIENT` + `CLEARANCE_CLUSTER_MISMATCH`
- New deny reasons specific to the clearance-mode check.
- Distinct from the existing `TAINT_ESCALATION` (runtime) and `SECRET_TO_EXTERNAL` (JADEPUFFER) so an AIOps layer can route them differently.

#### 4. `VerbPolicy.classification: ClassificationLevel = PUBLIC`
- New per-verb field recording the verb's highest sensitivity clearance.
- Default `PUBLIC` for backward compatibility with existing policies.

#### 5. `VerbPolicy.cluster: str = "default"`
- New per-verb field recording the verb's cluster. The cluster is a *partition key* for the policy registry — verbs in the same cluster are mutually clearable.
- Default `"default"` for backward compatibility.

#### 6. `ChainVerdict` DSCC fields
- `chain_classification: ClassificationLevel` — the C_eff of the chain.
- `cluster_set: Tuple[str, ...]` — the sorted set of cluster names that appeared in the chain.
- `clearance_violations: List[Dict[str, Any]]` — per-violation details (verb_name, policy_classification, required_classification, cluster).
- `composition_mode: Optional[CompositionMode]` — the mode that produced this verdict.
- All 4 fields preserved through `to_dict()`.

#### 7. `gate.set_composition_mode(mode)` + `gate.composition_mode` property
- Runtime mode switching (the paper fixes mode at construction, but the substrate allows post-construction override for A/B testing).

#### 8. `gate.register_cluster(verb_name, cluster)` + `gate.lookup_cluster(verb_name)` + `gate.clusters` property
- Cluster registry. `register_cluster` adds a verb to a cluster; `lookup_cluster` returns the cluster name (or `"default"` if not set).

#### 9. `gate.chain_effective_classification(chain)` — the high-water mark
- Returns `max(initial_classification, max(tool.classification for tool in chain))`.
- Monotonic by construction.
- An empty chain returns `initial_classification` (defaulting to PUBLIC).

#### 10. `gate.mrs_clearance_check(chain)` — the Phase 1 MRS check
- Returns `(ok, failing_verbs, c_eff, failing_clusters)`.
- For each step, checks if `policy.classification >= c_eff`. If not, the verb is in `failing_verbs` and its cluster is in `failing_clusters`.
- This is the *static* check; the runtime taint tracking is unaffected.

#### 11. `gate.cluster_compatible(a, b)` + `gate.chain_cluster_compatibility(chain)`
- `cluster_compatible` returns True if two clusters can coexist in a chain (same cluster or one is `"default"`).
- `chain_cluster_compatibility` returns `(ok, conflicts)` where `conflicts` is the list of `(cluster_a, cluster_b)` pairs that violate the rule.

#### 12. Clearance check in `check_chain()`
- After the existing per-step taint tracking, the gate computes `chain_cls, violations, cluster_seen` and:
  - In CLEARANCE mode with violations: append `DenyReason.CLEARANCE_LEVEL_INSUFFICIENT` (or `CLEARANCE_CLUSTER_MISMATCH` if cluster names differ) and force the action to `BLOCK_AND_ESCALATE`.
  - In CLEARANCE mode without violations: chain proceeds (existing action is preserved).
  - In TAINT mode: clearance check is skipped; `chain_cls` and `cluster_seen` are still recorded for audit.
- The verdict's `chain_classification`, `cluster_set`, `clearance_violations`, and `composition_mode` are populated regardless of mode.

#### 13. `create_gate()` updated
- New constructor params: `composition_mode=CompositionMode.CLEARANCE`, `initial_classification=None` (defaults to PUBLIC), `clusters=None` (default cluster registry).
- `create_jadepuffer_demo_gate()` updated: accepts `composition_mode=` and passes it through; the JADEPUFFER demo now defaults to CLEARANCE mode (matching the paper's default).

#### 14. `cli/dscc_clearance.py` — operator-facing entry point
- Reads `--policies` and `--chain` JSON files (same format as `compositional_review`).
- Optional `--mode=clearance|taint` (default clearance), `--initial=public|internal|confidential|restricted` (default public), `--clusters` JSON file.
- Prints the verdict, C_eff, mode, and violation count.
- JSON output (`--json`) returns the full `ChainVerdict.to_dict()`.
- Exit codes: 0 = ALLOW, 1 = BLOCK / BLOCK_AND_ESCALATE.

## Real Findings Surfaced

1. **The default mode change is *not* a backward-compat break.** The original `check_chain` was implicitly in TAINT-like mode (no clearance check). Today's run makes the default CLEARANCE — a *stricter* default. All existing tests pass because the JADEPUFFER-pattern chains in existing tests are either already in the same cluster (so cluster_compatible returns True) or the chain is short and within a single classification level. The substrate's existing test suite inadvertently exercises the *intersection* of the runtime taint tracking (which it was designed for) and the static clearance check (which it now gets for free). A future test pass should add *mixed-classification* regression tests to lock in the clearance-mode default.
2. **The `mrs_clearance_check` returns 4-tuple, not 3-tuple.** Initial implementation returned `(ok, failing, c_eff)`; the failure-test that expected `failing_cls` to contain the policy's classification-level value (not its cluster name) caught a *semantic ambiguity* in the field name. Resolved by: `failing_cls` returns the *cluster names* (the strings in the `clusters` map), and a separate `clearance_violations` field on the verdict holds the per-violation `Dict` (which contains both the policy's classification and the cluster). The 4-tuple return is the *lightweight* version; the verdict's `clearance_violations` is the *rich* version.
3. **The JADEPUFFER chain already tripped the clearance check in CLEARANCE mode.** The JADEPUFFER demo gate registers `read_env` (INTERNAL), `read_secret_store` (RESTRICTED), and `http_post_external` (PUBLIC). The chain's C_eff is RESTRICTED (max of the three), and `http_post_external` is PUBLIC — *under-classified*. The chain now trips both `SECRET_TO_EXTERNAL` (runtime) AND `CLEARANCE_LEVEL_INSUFFICIENT` (static). The JADEPUFFER attack is now caught *twice*: by the runtime taint tracking (which existed) and by the static clearance check (which is new).

## Research Synthesis

- **The substrate now implements *both* DSCC phases.** Phase 1 (MRS / static clearance check) is the new `mrs_clearance_check` + `cluster_compatible` machinery. Phase 2 (runtime taint tracking) is the existing `check_chain` per-step taint logic. Together: the substrate catches unsafe chains *before* any tool runs (Phase 1) AND catches runtime data-policy violations (Phase 2). The two phases are *defense in depth* — the substrate's `CompositionMode.CLEARANCE` is the *strict* setting, `CompositionMode.TAINT` is the *permissive* setting.
- **The DSCC paper's 79.2%/95.5% block rate is a *property of the policy registry*, not the gate.** The block rate is `1 - (fraction of policy pairs/triples whose classifications are compatible and whose clusters match)`. A well-designed policy registry (e.g., all INTERNAL tools in the "internal" cluster, all RESTRICTED tools in the "restricted" cluster) gives a high block rate. The gate is the *enforcement substrate*; the operator's policy registry is the *policy substrate*.
- **The JADEPUFFER attack is now caught *twice*.** Static check (clearance mode): `read_env` (INTERNAL) → `http_post_external` (PUBLIC) is *under-classified* (C_eff is RESTRICTED after `read_secret_store`); the chain is rejected before any tool runs. Runtime check (existing taint tracking): the chain's `SECRET → EXTERNAL` data flow trips `DenyReason.SECRET_TO_EXTERNAL`. The substrate's defense-in-depth is now *visible* in the verdict's `clearance_violations` + `reasons` fields.
- **The `composition_mode` is a per-gate switch, not a per-chain switch.** The paper fixes mode at construction. The substrate allows runtime override via `set_composition_mode()` for A/B testing and policy iteration. In production, the mode should be set at gate construction and not changed at runtime.
- **The substrate's `cluster` field is *operator-supplied*, not derived.** The operator decides which verbs go in which cluster. The substrate doesn't *infer* clusters (that's a separate research problem — clustering policy registries by semantic equivalence). The substrate *enforces* the operator's cluster assignment. This matches the DSCC paper's model: every tool carries a security policy *authored by the tool developer or organizational security team*.
